# Vendors — authoritative external sources

The canonical record of Lattice's load-bearing third-party dependencies, **where their authoritative
behavior is documented**, and **which version we pin**. Referenced from `CLAUDE.md` ("Authoritative
external sources").

**Rule.** When you need the authoritative behavior of a vendored dependency — semantics, version-gated
features, edge cases — cite the **upstream project's own docs / source / ADRs, version-matched to our
pinned version**. Never rely on a secondary blog or an unqualified web search. Web search is a last
resort and must be **corroborated against the upstream** (the project's docs site or its source/ADRs at
the pinned version) before you act on it.

Add a row when a new vendor's behavior becomes load-bearing.

| **IndexedDB** (W3C/WHATWG) | The **browser** Edge node's Local VAL Store (`internal/edge/store/idb.go`, edge-browser-node-design.md §3.3) — the bbolt store's browser counterpart, and the only durable, transactional, origin-scoped, Worker-available storage a browser offers. Two semantics are load-bearing and non-obvious: (1) **a transaction is active only while a request it issued is pending**, and auto-commits once control returns to the event loop with none outstanding — so a read-modify-write (the last-writer-wins gate) must issue its write from *inside* the read's success callback, never after a Go channel round-trip; (2) a **key generator** (`autoIncrement`) is monotonic and persists with the object store, which is what makes it a faithful stand-in for bbolt's `NextSequence` for the intent queue. Transactions with overlapping scope run in creation order, which is what serialises concurrent deltas. | <https://www.w3.org/TR/IndexedDB/> — the W3C spec is the authority (transaction lifetime + "active" flag; key generators; `blocked` on version change). MDN is a readable secondary, not the citation. | No pin — a browser platform API. The behaviour is pinned *by test* instead: `make test-edge-idb-conformance` runs the store conformance suite against a real IndexedDB in headless Chrome. |
| **agnivade/wasmbrowsertest** | **Test-only** `go test -exec` runner that executes a js/wasm test binary in a real headless Chrome, giving `internal/edge/store`'s conformance suite a **real IndexedDB** (`make test-edge-idb-conformance`, CI job `edge-browser-store`). Chosen over Node + `fake-indexeddb` because Node ships no IndexedDB at all, the repo has no npm toolchain, and a fake would make a reimplementation of IndexedDB — not the engine the PWA actually runs on — the thing our semantics are proven against. | <https://github.com/agnivade/wasmbrowsertest> — upstream source + README (the `go test -exec` protocol; it drives Chrome via chromedp and needs a Chrome/Chromium on PATH, which the CI runner image ships). | **v0.11.0** — pinned by `WASM_BROWSER_TEST_VERSION` in the `Makefile`; installed on demand by the target, not a `go.mod` dependency (it is a tool, and never linked into a Lattice binary). |
| **Caddy** | The hosted demo's **single public surface**: TLS termination + reverse proxy in front of the loopback-bound apps (`deploy/demo/Caddyfile`, installed by `demo-bootstrap.sh`; Loupe joins at public launch — F20.4). Load-bearing defaults: incoming `X-Forwarded-*` from an untrusted client is **ignored** (anti-spoofing) and `X-Forwarded-For` is set from the immediate peer — why Loupe's credential-exchange limiter may trust the *last* XFF entry under the single-hop demo topology (loupe-f20 §6.5); `Host` passes through unchanged. The HTTP **rate-limit handler is NOT in standard builds** (third-party `mholt/caddy-ratelimit`, xcaddy-only) — why that limiter lives in Loupe, not the Caddyfile. | <https://caddyserver.com/docs> (esp. `caddyfile/directives/reverse_proxy`, header defaults) · <https://github.com/caddyserver/caddy>. | No version pin — installed from the official Caddy stable apt repo by `deploy/demo/demo-bootstrap.sh` (latest v2 stable at install time); re-verify header defaults against the installed version if a behavior becomes newly load-bearing. |
| **nats.js** (`@nats-io/nats-core` + `@nats-io/jetstream`) | The **browser** Edge node's NATS transport (`internal/edge/browser/shell`, edge-browser-node-design.md §3.3) — the browser counterpart to `nats.go`, which has **no js/wasm/browser transport** (issues `nats.go` #530/#588/#661, open since 2021; it dials raw TCP a browser cannot). `nats-core`'s `wsconnect` opens the W3C-WebSocket connection with token auth + `inboxPrefix`; `jetstream` layers the durable pull consumer. ## Version-gated NATS features (why the pin matters)

Feature availability is version-gated; cite the version that introduced a feature and confirm it is
≤ our pin:

| Feature | Introduced | Notes |
|---------|-----------|-------|
| Per-key message TTL (ADR-48) | NATS 2.11 | Idempotency-tracker 24h TTL (Contract #4 §4.3). |
| Atomic batch (single-stream multi-key, revision-conditioned) | NATS 2.12 | The Processor step-8 commit; the reason all Core KV is one stream. |
| `@at` one-shot message schedules (ADR-51) | NATS 2.12 | The temporal lane's freshness timers + the bridge poll/timeout lane (Contract #10 §10.4). |
| `@every` / 6-field cron / timezone message schedules (ADR-51) | NATS 2.14 | Recurring schedules — the cron-killer (Contract #10 §10.4 "Recurring schedules"). |

**Platform floor: NATS 2.14** (the highest of the above). Pinned in `go.mod` + `docker-compose.yml`;
recorded in Contract #4 §4.3. Do not assume a lower floor — `@every`/cron need 2.14 and the platform
provides it.
